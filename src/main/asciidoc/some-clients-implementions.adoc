对于需要通过网络传输进行交互的服务，都有一个称之为客户端或者驱动(driver)的来实现不同服务的应用层的协议解析处理。比如MySql，Couchbase， Redis， MongoDB， Memcached和MQ等等都有对应的驱动或者各种语言实现的客户端。既然是基于网络的，那么连接池可能是必备的组成部分。

== 客户端设计的逻辑
客户端的最主要的职责是实现其对应的服务端所涉及的应用层协议。除此之外，互联网应用基本上对应了集群的环境，意味着服务端一般来讲都会有多台机器组成的。那么客户端的另外一个关键因素就是基于每次请求，该发送给那些机器。在目标需求是说服务快速稳定的前提下，这块实现的好坏直接决定了你的睡眠质量。

=== Redis的实现逻辑
Redis 2.x系列的Java客户端的实现基本上只是负责Redis协议的解析执行。他实现的时候也不关心线程安全性，认为这是开发着的职责；也不关心网络错误，汇报必要的错误，留给开发者处理。同时，也不能感知服务端的拓扑变化。

=== MongoDB的实现逻辑
MongoDB的实现相比Redis则考虑的因素变得非常多了。客户端知道整个服务拓扑结构，还实现了基于统计的服务端节点选择器。比如统计响应时间（RTT）最短的服务器来发送请求。这时候的服务节点不仅仅是一个IP信息，而且包含了更多的附件属性，如tagSet，角色等给了客户端更多的数据供决策。

=== Cassandra
Cassandra的客户端不仅仅能知道整个服务的拓扑结构，而且也能知道一部分数据结构。这当然是和Cassandra整个服务的设计有关，但是这个客户端基本上能做的事情非常多。
